复杂的 XML 结构处理（尤其是涉及文献、档案类具有多层嵌套或特定标记的文档）以及文本关键词提取，最容易在**边界条件处理、性能损耗和提取精准度**上出现优化空间。

以下是为你量身定制的深度审查提示词（Prompt），你可以在切换到专属分支后，直接在 Claude Code 的交互界面中依次发给它：

### 第一步：审查 Skill 1 —— XML 文档解析与目录构建

针对这个 skill，重点在于检查它对**嵌套标签的处理**、**异常处理的鲁棒性**以及**大文件内存占用**。

**你可以对 Claude Code 这样说：**

> 我当前处于 `claude-code-review` 分支。请帮我重点审查负责 **“读取 XML、构建文章目录并提取正文”** 的这个 skill 模块（文件路径：`[填写你的文件路径，例如 src/skills/xml_parser.py]`）。
> 
> 请从以下几个维度进行深度扫描并给出具体的修改建议或重构代码：
> 
> 1. **结构鲁棒性：** 如果 XML 文档存在深度嵌套、标签不规范、或者缺少某些预期节点（如标题或正文标签缺失），当前代码能否优雅降级而不崩溃？
>     
> 2. **清洗彻底性：** 提取正文时，是否能完美剥离 XML 内联的标记/属性（如注释、特殊修饰标签），同时保留正确的文本换行和段落结构？
>     
> 3. **内存与性能：** 如果处理超大体积的 XML 文档，当前使用的解析库和遍历方式（如是一次性载入还是流式解析）是否存在内存泄漏或性能瓶颈？是否有必要更换为更高效的解析流（如 `lxml.etree.iterparse`）？
>     
> 4. **目录构建逻辑：** 构建文章目录的树状结构或线性列表的算法是否可以更简洁高效？
>     
> 
> 请先列出你发现的问题和优化思路，在得到我确认后再修改代码。

### 第二步：审查 Skill 2 —— 文章关键词提取

针对这个 skill，重点在于检查**提取算法的准确性**、**领域词汇的适应能力**以及**噪音过滤**。

**在处理完第一个 skill 并提交（commit）后，对 Claude Code 这样说：**

> 接下来，请帮我审查负责 **“为文章提取关键词”** 的这个 skill 模块（文件路径：`[填写你的文件路径]`）。
> 
> 请帮我检查并优化以下方向：
> 
> 1. **噪音过滤：** 当前的停用词（Stop words）过滤机制是否足够完善？是否会把一些高频但无实际意义的虚词、标点或结构性词汇误作为关键词提取出来？
>     
> 2. **领域自适应：** 针对特定领域的文章（例如历史、文献、专业学术档案），当前的提取逻辑（无论是基于 TF-IDF、TextRank 还是传统的统计规则）是否支持动态扩展领域词典或调整权重，以防止专业核心词被低估？
>     
> 3. **多词组/短语识别：** 当前算法是否只能提取单个词？能否识别出更有语义价值的复合短语（Noun Phrases）作为关键词？
>     
> 4. **代码可扩展性：** 如果未来需要将此处的本地提取算法升级为基于 LLM（大语言模型）的语义提取，当前函数的接口设计和数据流是否足够解耦、方便扩展？
>     
> 
> 请给出具体的代码审查报告和优化后的重构版本。

### 💡 落地操作小贴士

1. **善用 `Claude Code` 的交互特性：** Claude Code 允许你直接在终端中进行交互。当它给出修改方案时，如果发现某处改动违背了你原本的设计初衷，可以直接在终端回复它：“_第 X 行的改动不对，因为这个 XML 的结构在实际数据中可能会演变成……，请重新考虑。_”
    
2. **测试驱动：** 在让 Claude 修改完一个 skill 后，**务必在当前分支运行一下你原有的测试脚本或 Demo**。确保提取出的文章目录和关键词依然符合预期，再执行 `git add` 和 `git commit`。
    
3. **对比代码：** 在最终准备 `git merge` 回主分支前，可以在主分支执行 `git diff main claude-code-review`，做最后一次肉眼核对，看看 AI 帮你把代码精简和优化了多少。
